# agent-memory-ops — ハブ型エージェントメモリの運用

## アーキテクチャ

Claude Code の自動メモリ（`~/.claude/projects/<project>/memory/`）は素のままでは
端末ローカルで、同期も定期整理もされない。この運用型では:

- **実体をハブリポジトリ内に置く**: ハブワークスペース（オーケストレーションの起点と
  なるリポジトリ）直下の `agent-memory/` を実体とし、Claude Code 側のパスから
  シンボリックリンクで接続する。
- **同期はハブの既存インフラに乗せる**: ハブリポジトリの自動コミット・プッシュ・プル
  （フック等）がそのままメモリを端末間同期する。専用の同期機構は作らない。
- **新端末への配布は chezmoi run_once**: dotfiles の
  `run_once_after_link-claude-agent-memory.sh` がハブごとのリンクを冪等に作成する。
  スクリプトはハブのリスト（`HUBS`）を持ち、ハブ追加＝リスト1行追加
  （run_once は内容変更で全端末再実行される）。
- **ハブ境界＝信頼境界**: ハブ（個人ワークスペース・チームワークスペース・クライアント
  ハブ）ごとにメモリは分離され、自動では跨がない。ハブを跨ぐ知識の移動は、月次GCの
  昇格判定という**意図的なゲート**だけを通す（無審査の自動同期にしない）。

## 規約

- メモリファイルの形式（frontmatter・`MEMORY.md` 索引）は Claude Code ハーネスの
  規約に従う。ハブ側の文書規約（OKF 等）は **適用外**とし、ハブのリンタがあれば
  `agent-memory/` を除外設定に入れる。ディレクトリ内に `index.md` は置かない。
- ファイル名は kebab-case（ハイフン区切り）で統一する。
- **保護対象**: 他のスキル・ルールが名指しで参照するメモリ（例: ブリーフ正本ストアの
  所在解決記録 `creative-brief-stores`）は、GCで統合・削除しない。

## 月次GC手順

ハブのタスク管理に月次タスクとして登録し、期限が来たら実行する。

1. **重複・陳腐化の整理**: 内容が重複するメモリを統合し、事実が変わって古くなった
   メモリを削除・修正する。
2. **命名の正規化**: kebab-case に揃え、リネーム時は参照箇所（`MEMORY.md`・
   wikilink）も更新する。
3. **索引整合**: `MEMORY.md` の行と実ファイルの過不足を機械的に突き合わせる
   （`diff <(ls) <(索引から抽出)` 等）。
4. **正本への圧縮**: ハブ内の正本ドキュメント（`ideas/`・wiki 等）と内容が重複する
   メモリは、正本への参照1行＋要点に圧縮する。複製を残さない。
5. **ハブ跨ぎ知識の昇格判定**: `feedback-assetization` スキルが定義する
   「暗黙の受け入れ基準の資産化」の型（一次データ→抽出→階層判定→焼き付け）を、
   エージェントの知識一般に適用する工程。メモリを知識の最下層の一次データとみなし、
   各メモリに昇格テスト「**このハブの外のセッションでも生きる知識か**」を問い、
   生きるものは性質で昇格先を選ぶ:
   - 全セッションで常時従うべき運用ルール → **グローバル CLAUDE.md**（chezmoi 配布）。
   - 特定タスク種別の手順・ノウハウ → **該当するグローバルスキル**
     （既存スキルへの追記を新設より優先）。
   - 主体・チーム固有の内容知識 → **主体の正本ストア**（チーム wiki・`ideas/` 等）。

   昇格の作法は型と共通:
   - 昇格する知識には**根拠となった実例**（どの案件・どのセッションの何に由来するか）を添える。
   - 由来した回数（n）を正直に保つ。迷うもの（n=1 で再現未確認など）は昇格させず残し、
     次回のGCで再判定する（**2回目に出会った知識が昇格の有力候補**）。
   - 昇格済みの知識に**反例**が出たら、削除ではなく適用条件を絞るか元の階層へ降格する。
   - 昇格後のメモリは項目4と同様に正本への参照へ圧縮する（正本と複製の一方向規約）。
   - 共有領域（チーム wiki 等）へ昇格させる際は機密境界を確認する
     （個人や他主体の情報を持ち込まない）。

完了したらハブのタスクを完了にし、翌月分の期限で再登録する。

## 新しいハブへの導入手順

対象ハブのリポジトリで、以下を順に行う（全体で冪等になるように）。

1. **実体の移設**: `mv ~/.claude/projects/<プロジェクトディレクトリ名>/memory <ハブ>/agent-memory`
   → `ln -s <ハブ>/agent-memory ~/.claude/projects/<プロジェクトディレクトリ名>/memory`。
   プロジェクトディレクトリ名はハブ絶対パスの `/` と `.` を `-` に置換したもの。
   メモリがまだ無いハブでは空ディレクトリで開始してよい。
2. **run_once スクリプトへの登録**: dotfiles（chezmoi ソース）の
   `run_once_after_link-claude-agent-memory.sh` の `HUBS` リストにハブのパスを1行追加し、
   コミット・プッシュする（chezmoi の Edit Flow に従う）。
3. **リンタ除外**: ハブに文書規約チェック（OKF 等）があれば `agent-memory/` を除外に追加。
4. **ハブ文書へのポインタ追記**: ハブの AGENTS.md（または CLAUDE.md）に薄い節を1つ追加する。
   書くのは「実体の場所とシンボリックリンク接続」「同期がハブの既存フックに乗ること」
   「運用の定義元はこのスキル（`agent-memory-ops`）であること」「ハブ固有の除外設定」
   のみ。**GC手順や規約の本文をハブ側に複製しない**（ドリフトの温床になる）。
5. **索引更新**: ハブにルート索引（`index.md` 等）があれば `agent-memory/` の一行を追加。
6. **月次GCタスクの登録**: ハブのタスク管理システムに月次GCタスクを追加する
   （手順参照先はこのスキル）。
7. **検証**: シンボリックリンクの解決、ハブのコミット・プッシュでメモリが同期されること、
   リンタが通ることを確認する。

## トラブルシュート

- 新端末でメモリが見えない → ハブリポジトリのクローンと `chezmoi apply`（run_once）の
  実行順を確認。スクリプトはハブ未クローンでもリンクだけ先に張る（クローン後に有効化）。
- リンク先に実体ディレクトリが既にある → run_once スクリプトは自動マージせず
  タイムスタンプ付きで退避する。退避分は手動でマージする。
- ハブのリポジトリ改名・移動 → プロジェクトディレクトリ名が変わるため、旧パスの
  メモリが孤児化する。旧 `~/.claude/projects/<旧名>/memory` を確認し、必要分を
  移してから削除する。
